# 00 Overview: 開発全体の進め方

このステップでは、まだコードを書きません。何を作るのか、何を後回しにするのか、どう確認するのかを決めます。

## 作るもの

Research Note は、研究資料を登録して、その資料に基づいて質問へ回答する AstrBot プラグインです。

最小版では、以下だけを作ります。

- 資料を追加する。
- 資料を一覧表示する。
- 資料を検索する。
- 検索した資料を LLM に渡して回答する。
- 資料を削除する。

## まだ作らないもの

最初は以下を作りません。

- PDF 読み込み。
- URL 読み込み。
- Web 管理画面。
- Milvus。
- embedding 検索。
- 複数ユーザーごとの権限管理。
- NotebookLM のような高度な引用 UI。

理由は、最初に必要なのは「RAG の流れを理解すること」だからです。RAG の流れは、入力、保存、検索、プロンプト、出力の5つです。

```text
入力: ユーザーのコマンドや質問
保存: 資料を JSON に保存
検索: 質問に関連する資料を選ぶ
プロンプト: 選んだ資料を LLM に渡す
出力: 回答と使用資料を返す
```

## 1ステップごとの進め方

毎回この順番で進めます。

1. 変更するファイルを1つか2つに絞る。
2. 小さい変更を入れる。
3. AstrBot を再起動する。
4. コマンドを手で試す。
5. エラーが出たらログを見る。
6. 動いたら `git diff` を見る。
7. 次のステップへ進む。

## 完成の判断基準

開発では「コードを書いた」ではなく「動作確認した」を完成とします。

たとえば `/research help` を作るステップなら、完成条件は以下です。

- AstrBot が起動する。
- プラグイン読み込みエラーがない。
- チャットで `/research help` を送る。
- 期待した説明文が返る。

## エラー時の判断

エラーが出たら、まず以下を見ます。

- 一番下のエラー名。
- `File "...", line ...` のファイルと行番号。
- 自分が直前に変更した場所。

よくある原因は以下です。

- `import` を忘れている。
- インデントがずれている。
- コマンド関数の引数が間違っている。
- `yield` と `return` の使い方が混ざっている。
- 非同期関数に `await` を付け忘れている。

## このステップの完了条件

- 作る最小機能を説明できる。
- 最初は JSON 保存とキーワード検索でよいと理解している。
- AstrBot の起動コマンドが分かる。
- `git status` と `git diff` の用途が分かる。

## 開発者として最初に身につける見方

開発者は、コードを「上から全部暗記するもの」として見ません。コードを役割で分けて見ます。

Research Note では、以下の5つに分けて考えます。

```text
入口: AstrBot がどの関数を呼ぶか
入力: ユーザーが何を送ったか
状態: 何をファイルやメモリに保存しているか
処理: 検索、整形、LLM 呼び出しで何をしているか
出力: ユーザーへ何を返すか
```

たとえば `/research ask Transformer とは？` なら、開発者はこう読みます。

```text
入口: research_ask が呼ばれる
入力: question = "Transformer とは？"
状態: JSON に保存済みの資料を読む
処理: 関連資料を探し、プロンプトを作り、LLM に渡す
出力: 回答テキストと使用資料IDを返す
```

この見方ができると、大きなコードでも迷いにくくなります。

## アプリ開発とデータ分析の違い

データ分析では、ノートブック上で上から順番に実行することが多いです。

アプリ開発では、コードは上から順番に全部実行されるわけではありません。AstrBot が必要なタイミングで関数を呼びます。

たとえば以下です。

- 起動時に `__init__` が呼ばれる。
- ユーザーが `/research help` を送ったときだけ `research_help` が呼ばれる。
- ユーザーが `/research ask ...` を送ったときだけ `research_ask` が呼ばれる。

つまり、開発者は「いつ、誰が、この関数を呼ぶのか」を常に考えます。

## ログとエラーは敵ではない

開発では、エラーは失敗ではなく情報です。

エラーが出たときは、まずこの順番で見ます。

1. 一番下のエラー名を見る。
2. 自分のプラグインのファイル名が出ている行を見る。
3. 直前に変更したコードを見る。
4. エラー文に出ている変数名や関数名を見る。

たとえば以下なら、`json` を import していない可能性が高いです。

```text
NameError: name 'json' is not defined
```

たとえば以下なら、関数の引数や型が合っていない可能性があります。

```text
TypeError: research_ask() missing 1 required positional argument: 'question'
```

本物の開発者は、エラーを見て原因候補を絞ります。エラーを怖がらず、読めるようになることが大事です。

## 小さく作る理由

最初から完成形を作らないのは、実力がないからではありません。プロの開発でも、小さく動くものを作ってから広げます。

小さく作るメリットです。

- どこで壊れたか分かりやすい。
- 動くたびに自信がつく。
- 後から設計を直しやすい。
- エラーが少ない。
- 学習が積み上がる。

Research Note では、最初の「小さく動くもの」は JSON 保存とキーワード検索です。これは幼稚な実装ではなく、RAG の骨格を理解するための正しい最短ルートです。
